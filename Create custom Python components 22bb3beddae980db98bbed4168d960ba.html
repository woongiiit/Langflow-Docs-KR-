Create custom Python components

커스텀 Python 컴포넌트 제작

: Components > Create custom Python components

1. 커스텀 컴포넌트란?

커스텀 컴포넌트(custom component)는 Langflow의 기능을 확장하기 위한 파이썬 클래스입니다.

모든 커스텀 컴포넌트는 Component 클래스를 상속해서 작성하며,

이를 통해 새로운 기능 추가, 데이터 처리, 외부 서비스 연동, 특수 도구 구현 등이 가능합니다.

Langflow의 노드 기반 환경에서는 각 노드가 컴포넌트로서 독립적인 기능을 수행합니다.

커스텀 컴포넌트는 아래와 같은 세 가지 구성 요소를 정의하는 파이썬 클래스입니다.

  • 입력(Inputs): 컴포넌트가 동작하기 위해 필요한 데이터 또는 매개변수
  • 출력(Outputs): 컴포넌트가 처리 후 다음 노드에 제공하는 데이터
  • 로직(Logic): 입력을 받아 출력을 생성하는 내부 처리 방식

커스텀 컴포넌트를 만들면 다음과 같은 장점이 있습니다.

  • 기능 무한 확장
  • 재사용성
  • 입력 값에 따른 자동 UI 필드 생성
  • 노드 간 타입 안전 연결

커스텀 컴포넌트는 특화된 작업, 외부 API 호출, 고급 로직 추가 등에 사용됩니다.

1. 커스텀 컴포넌트의 기본 구조

Langflow에서 커스텀 컴포넌트는 다음 네 가지를 기반으로 작성됩니다.

  1. Component 클래스를 상속하는 파이썬 클래스
  1. 컴포넌트를 식별·설명하는 클래스 단위 속성
  1. 데이터 흐름을 정하는 입력/출력 목록
  1. 로깅 및 고급 로직을 위한 내부 변수

클래스 단위 속성(Class-level attributes)

아래와 같이 클래스 단위 속성을 정의해서 컴포넌트의 UI 표시 및 동작을 제어합니다.

class MyCsvReader(Component):
    display_name = "CSV Reader"      # 노드 헤더에 표시될 이름
    description = "Reads CSV files"  # 툴팁 설명
    icon = "file-text"               # 아이콘 (시각적 식별자)
    name = "CSVReader"               # 내부 고유 ID
    documentation = "http://docs.example.com/csv_reader"  # (선택) 외부 문서 링크
  • display_name: 노드 헤더에 표시될 사용자 친화적 이름
  • description: 툴팁으로 표시되는 짧은 설명
  • icon: Langflow 아이콘 라이브러리에서 선택하는 시각적 식별자
  • name: 내부적으로 고유하게 식별하는 값
  • documentation: (선택) 외부 문서 링크

아이콘 사용법

Langflow는 Lucide 아이콘 라이브러리를 사용합니다.

예를 들어, icon = "file-text"와 같이 문자열로 지정하면

자동으로 Lucide의 해당 아이콘이 렌더링됩니다.

2. 커스텀 컴포넌트의 구조

커스텀 컴포넌트는 단순히 입력/출력만 있는 클래스가 아닙니다.

내부적으로 라이프사이클, 출력 생성, 프론트엔드 상호작용, 로직 조직화 등의 구조를 가집니다.

기본 예시

  • langflow.custom.Component 상속
  • display_name, description, icon 등 메타데이터 선언
  • 입력/출력 목록 정의
  • 출력과 매칭되는 메소드 구현

최소 예시 코드

from langflow.custom import Component
from langflow.template import Output

class MyComponent(Component):
    display_name = "My Component"
    description = "A short summary."
    icon = "sparkles"
    name = "MyComponent"

    inputs = []
    outputs = []

    def some_output_method(self):
        return ...

내부 라이프사이클 및 실행 흐름

Langflow 엔진이 컴포넌트의 다음과 같은 라이프사이클을 관리합니다.

  1. 생성(Instantiation): 컴포넌트 객체 생성 및 내부 구조 초기화
  1. 입력 할당(Assigning Inputs): UI나 연결을 통해 입력값 할당
  1. 유효성 검사 및 설정(Validation and Setup): 필요 시 _pre_run_setup 등 실행 전 준비 로직 수행
  1. 출력 생성(Outputs Generation): run() 또는 build_results()가 출력 메소드 실행

선택적 훅(Hook)

  • initialize_data 또는 _pre_run_setup으로 실행 전 초기화 로직 구현 가능
  • __call__, run(), _run()을 오버라이드하면 호출/실행 방식 직접 지정 가능

3. 입력과 출력 정의

커스텀 컴포넌트의 입력(Inputs), 출력(Outputs)은 각각 아래 속성으로 정의합니다.

입력(Inputs)

입력 목록(inputs)은 클래스 단위에서 선언하며,

Langflow가 컴포넌트를 불러올 때 이 목록을 참고해 UI 필드와 포트를 렌더링합니다.

입력은 보통 langflow.io의 클래스 인스턴스(예: StrInput, DataInput, MessageTextInput)로 정의합니다.

주요 파라미터

  • name: 내부 변수명(self.<name>으로 접근)
  • display_name: UI에 표시되는 이름
  • info (선택): 툴팁 또는 설명
  • value (선택): 기본값
  • advanced (선택): 고급 설정 영역으로 분리
  • required (선택): 필수값 여부
  • is_list (선택): 여러 값 허용
  • input_types (선택): 연결 가능 타입 제한(예: ["Data"])

주요 입력 클래스 예시

  • 텍스트 입력
    • StrInput: 한 줄 텍스트
    • MultilineInput: 여러 줄 텍스트
  • 숫자/불리언 입력
    • BoolInput, IntInput, FloatInput
  • 드롭다운 선택
    • DropdownInput
  • 시크릿 입력
    • SecretStrInput (API 키 등)
  • 데이터/메시지 입력
    • DataInput, MessageInput, MessageTextInput
  • 핸들 입력
    • HandleInput
  • 파일 업로드
    • FileInput
  • 리스트 입력
    • is_list=True 옵션 사용

입력 정의 예시

from langflow.io import StrInput, BoolInput, DropdownInput

inputs = [
    StrInput(name="title", display_name="Title"),
    BoolInput(name="enabled", display_name="Enabled", value=True),
    DropdownInput(name="mode", display_name="Mode", options=["Fast", "Safe", "Experimental"], value="Safe")
]

출력(Outputs)

출력 목록(outputs)은 클래스 단위에서 선언하며,

각 출력은 UI의 연결 포인트로 나타납니다.

출력에 연결이 이루어지면 Langflow는 자동으로 해당 메소드를 실행하고 반환값을 다음 컴포넌트로 전달합니다.

출력은 일반적으로 langflow.io의 Output 인스턴스로 정의합니다.

주요 파라미터

  • name: 내부 변수명
  • display_name: UI 표시 이름
  • method: 출력 생성에 사용될 메소드 이름
  • info (선택): 도움말 텍스트

출력 메소드는 클래스에 반드시 존재해야 하며, 반환 타입을 명시하는 것이 권장됩니다.

진행 상황이나 로그는 메소드 내에서 self.status 메시지로 출력할 수 있습니다.

주요 반환 타입

  • Message: 구조화된 채팅 메시지
  • Data: 유연한 데이터 객체(.data, .text 속성)
  • DataFrame: 판다스 기반 테이블
  • 기본 타입(str, int, bool 등): 타입/색상 일관성이 필요하면 권장하지 않음

출력 정의 예시

from langflow.custom import Component
from langflow.io import DataInput, Output
from langflow.schema import Data, DataFrame

class DataToDataFrame(Component):
    display_name = "Data to DataFrame"
    description = "Convert multiple Data objects into a DataFrame"
    icon = "table"
    name = "DataToDataFrame"

    inputs = [
        DataInput(
            name="items",
            display_name="Data Items",
            info="List of Data objects to convert",
            is_list=True
        )
    ]

    outputs = [
        Output(
            name="df_out",
            display_name="DataFrame Output",
            method="build_df"
        )
    ]

    def build_df(self) -> DataFrame:
        rows = []
        for item in self.items:
            row_dict = item.data.copy() if item.data else {}
            row_dict["text"] = item.get_text() or ""
            rows.append(row_dict)

        df = DataFrame(rows)
        self.status = f"Built DataFrame with {len(rows)} rows."
        return df

4. 타입 어노테이션(Typed annotations)

Langflow에서는 타입 어노테이션을 활용하여

사용자에게 시각적 안내를 제공하고, 플로우(연결 흐름)의 일관성을 유지할 수 있습니다.

타입 어노테이션의 장점은 다음과 같습니다.

  • 색상 구분: > Data, > Message와 같이 반환 타입에 따라 출력 포트의 색상이 다름
  • 유효성 검사: 호환되지 않는 연결은 Langflow가 자동으로 차단
  • 가독성 향상: 개발자가 데이터 흐름을 빠르게 이해 가능
  • 개발 도구 지원: 에디터에서 코드 자동완성, 오류 체크 기능 향상

자주 사용하는 반환 타입

Message

채팅형 출력에 적합

def produce_message(self) -> Message:
    return Message(text="Hello! from typed method!", sender="System")

UI 상에서는 Message 타입 입력과만 연결 가능

Data

딕셔너리, 텍스트 등 구조화된 데이터에 적합

def get_processed_data(self) -> Data:
    processed = {"key1": "value1", "key2": 123}
    return Data(data=processed)

UI 상에서는 DataInput과만 연결 가능

DataFrame

표(tabular) 형태 데이터에 적합

def build_df(self) -> DataFrame:
    pdf = pd.DataFrame({"A": [1, 2], "B": [3, 4]})
    return DataFrame(pdf)

UI 상에서는 DataFrameInput과만 연결 가능

기본 타입 (str, int, bool 등)

기본형 반환도 가능하나,

UI 일관성을 위해 Data, Message 등으로 감싸서 반환하는 것이 권장됩니다.

def compute_sum(self) -> int:
    return sum(self.numbers)

타입 어노테이션 사용 시 팁

  • 항상 반환 타입 명시: > Data, > Message, > DataFrame 등 명시하면 UI 색상 구분 및 유효성 검증이 제대로 동작
  • Raw 데이터 감싸기: 딕셔너리 등은 Data, Message, DataFrame 등으로 감싸서 반환
  • 기본형 사용 신중히: 단순 플로우에서는 str/int 반환도 허용되나, 감싸서 반환하는 것이 확장성에 유리
  • 헬퍼 함수에도 타입 명시: 내부 함수라도 typing을 명확히 하면 유지보수 용이
  • 에러 상황 구조화: 예외적 상황 발생 시 error 필드가 포함된 Data 객체 반환 권장
  • 타입 일관성 유지: 컴포넌트 전반에서 동일한 타입 사용 권장

5. 동적 필드(Dynamic fields) 활성화

동적 필드는 사용자의 선택에 따라 입력 항목이 변하거나, 새롭게 나타나거나 사라질 수 있도록 합니다.

dynamic=True로 입력을 선언하면 동적 필드가 활성화됩니다.

real_time_refresh=True를 함께 설정하면, 입력값이 바뀔 때마다 update_build_config 메서드가 호출되어

실시간으로 UI에 필드 표시 여부나 속성을 반영할 수 있습니다.

아래 예시는 operator 입력 필드가 바뀌면(특히 "regex" 선택 시)

regex_pattern 필드의 표시 여부를 제어하는 방식입니다.

from langflow.io import DropdownInput, StrInput

class RegexRouter(Component):
    display_name = "Regex Router"
    description = "Demonstrates dynamic fields for regex input."

    inputs = [
        DropdownInput(
            name="operator",
            display_name="Operator",
            options=["equals", "contains", "regex"],
            value="equals",
            real_time_refresh=True,
        ),
        StrInput(
            name="regex_pattern",
            display_name="Regex Pattern",
            info="Used if operator='regex'",
            dynamic=True,
            show=False,
        ),
    ]

update_build_config 구현

real_time_refresh=True가 적용된 필드의 값이 바뀌면

Langflow가 update_build_config 메서드를 호출하여

다른 입력 필드의 표시 여부·속성 등을 동적으로 조정할 수 있습니다.

아래는 사용자가 operator 필드를 "regex"로 변경하면 regex_pattern 필드를 표시하고,

그 외에는 숨기는 예시입니다.

def update_build_config(self, build_config: dict, field_value: str, field_name: str | None = None) -> dict:
    if field_name == "operator":
        if field_value == "regex":
            build_config["regex_pattern"]["show"] = True
        else:
            build_config["regex_pattern"]["show"] = False
    return build_config

동적 필드 속성 추가 제어

  • required: 필수 입력 여부 (build_config["some_field"]["required"] = True/False)
  • advanced: 고급 입력 여부 (build_config["some_field"]["advanced"] = True)
  • options: 드롭다운 옵션 동적 변경 등

동적 필드 관리 팁

  • 불필요한 필드는 최소한만 숨기기(과도한 변화는 사용자 혼란 유발)
  • 필드 추가/삭제 시 기존 입력값이 소실되지 않는지 반드시 테스트
  • 값 유지를 위해 show=False로만 숨기기(필드 삭제 X)
  • 필드가 나타나는 조건을 info에 설명문으로 명확히 추가
  • 동적 로직이 복잡해질 경우, 가능하다면 컴포넌트를 분리해서 관리

6. 에러 처리 및 로깅(Error handling and logging)

Langflow에서는 신뢰성 있는 컴포넌트 작동을 위해

다음과 같은 에러 처리 방식을 권장합니다.

에러 처리 방법

  1. 예외 발생(Python Exception 사용)
    • 치명적 오류 발생 시, ValueError 등 표준 예외 또는 ToolException 등을 사용해 에러 발생
    • Langflow가 예외를 자동으로 감지하여 UI에 에러 메시지 표시
    def compute_result(self) -> str:
    if not self.user_input:
        raise ValueError("No input provided.")
  1. 구조화된 에러 데이터 반환
    • 플로우 전체를 중단하지 않고, "error" 필드가 포함된 Data 객체 반환
    • 이후 단계 컴포넌트에서 에러 감지 및 처리 가능
    def run_model(self) -> Data:
  try:
      # ...
  except Exception as e:
      return Data(data={"error": str(e)})

디버깅 및 플로우 관리 개선

  • self.status 활용: 실행 결과, 성공/진행/에러 요약 등 간단한 메시지 저장

    UI에서 바로 확인 가능

    def parse_data(self) -> Data:
    # ...
    self.status = f"Parsed {len(rows)} rows successfully."
    return Data(data={"rows": rows})
  • 특정 출력 중단(self.stop):

    여러 출력 포트가 있는 컴포넌트에서 특정 조건 미충족 시

    해당 출력만 중단(전체 컴포넌트 중단 X)

    def some_output(self) -> Data:
    if <some condition>:
        self.stop("some_output")  # 해당 포트만 중단
        return Data(data={"error": "Condition not met"})
  • 로깅(self.log):

    실행 중 주요 이벤트를 기록

    로그는 컴포넌트 상세 화면의 "Logs" 또는 "Events" 섹션에서 확인 가능

    디버그 패널, 내보내기 기능 등으로 활용 가능

    def process_file(self, file_path: str):
    self.log(f"Processing file {file_path}")
    # ...

에러 처리 및 로깅 팁

  • 입력값 미리 검증(초기 오류 사전 차단)
  • self.status로 간결한 결과 요약 제공
  • 로그는 핵심 정보 위주로 관리(UI 혼잡 방지)
  • 구조화된 에러 반환 권장(Data(data={"error": ...}))
  • 필요한 경우에만 특정 출력 중단(self.stop), 전체 중단 최소화
